# Правила для — Создание веб-сайтов

---

## 0. Прояснение задачи — главный приоритет

Перед тем как писать любой код, ИИ **обязан** задать все необходимые вопросы.
Количество вопросов не ограничено — лучше спросить лишний раз, чем сделать
неправильно и переделывать.

**Когда задавать вопросы:**

- Задача сформулирована размыто или неполно
- Непонятен стек, который использует проект
- Неясна структура существующего проекта (если он уже есть)
- Не определён дизайн или визуальный стиль
- Есть несколько равнозначных способов реализации — нужен выбор
- Любое архитектурное решение, которое потом сложно изменить

**Формат уточнений:** Вопросы задаются списком, все сразу — не по одному после
каждого ответа. Каждый вопрос конкретный, с вариантами ответа там, где это
уместно.

## 1. Базовые принципы и UI/UX

**Качество интерфейса** Должно ощущаться премиальным, современным, свежим.
Интерфейс — отзывчивым (responsive) и дружелюбным (human copy / friendly
approachable).

**Производительность** Должно работать максимально быстро. Строгий запрет на
антипаттерны производительности (лишние ре-рендеры, блокирующие операции,
не-мемоизированные вычисления). Используй если надо React Compiler, Писать
чистый декларативный код без микрооптимизаций, пока не доказана проблема с
производительностью

**Web First** Проект ориентирован на браузер. Поведение, стили и анимации должны
учитывать веб-специфику: hover-эффекты уместны, курсор `pointer` — норма там,
где элемент кликабелен.

**Доступность (Accessibility)**

- Обязательная поддержка управления с клавиатуры (`tabIndex`, `onKeyDown`)
- Семантический HTML: `<nav>`, `<main>`, `<section>`, `<article>`, `<button>`
  вместо `<div>` везде, где есть смысл
- ARIA-атрибуты там, где семантики HTML недостаточно
- Контрастность текста — минимум WCAG AA

**SEO**

- Корректные мета-теги: `<title>`, `<meta name="description">`, `og:*`
- Семантическая структура заголовков: один `<h1>` на страницу, иерархия
  соблюдается
- Изображения всегда с атрибутом `alt`

## 2. Структура проекта и архитектура файлов

**Ограничение строк кода** Каждый файл — строго максимум 100–150 строк. Всё, что
больше — рефакторится и разбивается.

**Один компонент — один файл** Строгое соблюдение правила.

**Группировка по фичам**

**Алиасы для импортов** Обязательное использование алиасов (например,
`@/components/...`). Запрет на относительные пути вида `../../../`.

**Константы для роутинга** Пути страниц не хардкодятся. Используется объект
констант в `UPPER_CASE`

## 3. Паттерны написания кода

**Именование компонентов**

- Компоненты — в `PascalCase`
- Исключение: файлы роутинга Next.js (`page.tsx`, `layout.tsx`, `loading.tsx`)

**Экспорты** Избегать `export default`. Использовать именованные экспорты —
минимизирует риск циклических зависимостей. Исключение: страницы Next.js, где
дефолтный экспорт требуется фреймворком.

**Enums** Не использовать встроенные TypeScript `enum`. Вместо них —
объекты-константы

**Разделение логики в циклах** Если внутри `map` используются условия или
дочерние компоненты — выносить их в отдельный компонент или функцию. Запрет на
сложные конструкции прямо внутри цикла.

**Типизация**

- Запрет на `any`
- Запрет на нечитаемые аргументы (деструктуризация обязательна, где уместно)
- Явная типизация пропсов через `interface Props { ... }`

**Store (Zustand)** Глобальный стейт — дробить по функционалу. Один файл — одна
сущность:

**Async/Await** Запрет на `.then().catch()` цепочки там, где можно использовать
`async/await`. Обработка ошибок — через `try/catch`.

## 4. Стилизация, верстка и анимации

**Утилита cn()** Обязательное использование для объединения и условного рендера
Tailwind-классов

**Система отступов** Все `padding`, `margin`, `width`, `height` — кратны 4px (4,
8, 12, 16, 20, 24...). Tailwind-классы уже следуют этой сетке.

**Анимации (Framer Motion)**

- Использовать только GPU-accelerated свойства: `transform`, `opacity`
- Запрет на анимацию `box-shadow`, `width`, `height`, `top`, `left` — браузер
  плохо это переваривает
- Анимации должны быть плавными и уместными, не раздражающими

**Иконки**

- Основная библиотека: `lucide-react`
- Брендовые иконки: специализированные библиотеки (например, `react-icons/si`)
- Запрет на ручную отрисовку SVG для иконок

**Темизация** Поддержка светлой и тёмной темы. Использовать CSS-переменные или
Tailwind `dark:` модификатор. Цвета не хардкодятся в виде hex напрямую — только
через токены.

**Изображения**

- Всегда использовать `<Image>` из `next/image` вместо `<img>`
- Обязательные атрибуты: `alt`, `width`, `height` или `fill`
- Форматы: предпочтительно WebP/AVIF
- Проект использует Tailwind CSS v4. Все кастомные токены, цвета и шрифты
  задаются через CSS-переменные (@theme / @layer theme), а не через
  tailwind.config.js

## 5. Работа с данными

**Запросы к API**

- Все запросы — через TanStack Query (`useQuery`, `useMutation`)
- Состояния загрузки и ошибок обрабатываются явно (скелетоны, fallback UI)
- Запрет на прямые `fetch` внутри компонентов без абстракции Мутации и обработка
  форм должны выполняться через Server Actions (с использованием хуков React 19:
  useActionState, useFormStatus). TanStack Query использовать преимущественно
  для сложного клиентского дата-фетчинга и синхронизации стейта. Использовать
  хук use() для резолва промисов на клиенте там, где это уместно

**Серверные компоненты (Next.js App Router)**

- Данные, не требующие интерактивности, — в Server Components
- Клиентские компоненты (`'use client'`) — только там, где нужны хуки, события
  или стейт

**Переменные окружения**

- Секреты — только в `.env.local`, никогда в коде
- Публичные переменные — `NEXT_PUBLIC_` префикс
- Типизировать env через `env.ts` с zod-валидацией

## 6. Качество кода и запреты

**Запрещено:**

- `any` в TypeScript
- `console.log` в продакшн-коде (только `console.error` для ошибок)
- Хардкод строк: URL, конфиги, тексты интерфейса — в константы или i18n
- Инлайн-стили через `style={{}}`, если то же достигается Tailwind-классом
- Мутация стейта напрямую (только через сеттеры)
- `useEffect` для логики, которую можно вынести в обработчик события

**Обязательно:**

- `key` в списках — уникальный идентификатор из данных, не индекс массива
- Обработка пустых состояний (empty state) и состояний ошибок
- Логические операции: предпочитать `??` и `?.` вместо `||` и ручных проверок на
  `undefined`
